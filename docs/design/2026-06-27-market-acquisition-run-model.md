# Market Acquisition Run Model

## Purpose

Market Acquisition has grown from a probe into an execution pipeline. The current code can stage a dashboard request, claim it in the plugin, prepare an advisory route, travel through worlds, search the market board, validate live rows, and purchase safe listings. The next source of risk is no longer whether the pieces can work; it is whether every piece agrees about what run is active, what state is authoritative, and which failures are recoverable.

This document formalizes the run model so the dashboard, server lifecycle, plugin UI, route runner, diagnostics, and purchase loop stop sharing implicit state.

## Goals

- Give every local execution attempt a stable attempt identity under the dashboard request id.
- Separate server request lifecycle from plugin route execution lifecycle.
- Make progress reporting idempotent without hiding real conflicts.
- Make plugin reloads, route restarts, failed runs, and stale server state recoverable.
- Suppress irrelevant warnings during expected transition states such as world travel, character unload, and market-board window churn.
- Keep live in-game market-board rows as the purchase authority.
- Keep the dashboard as an intent and observability surface, not a remote-control surface.
- Make diagnostics persistent and detailed enough to explain server, route, and UI automation failures after the fact.

## Non-Goals

- Multiple simultaneous active acquisition attempts in one plugin instance.
- Server-owned execution of game UI automation.
- Background polling from the plugin when the user has not initiated pickup or execution.
- Treating remote market data as purchase authority.
- Hiding catastrophic or ambiguous failures behind automatic retries.
- Requiring routine user input after route launch, except for the current temporary per-world confirmation/testing affordance if still enabled.

## Core Concepts

### Acquisition Request

An acquisition request is the dashboard-created intent stored on the server.

It answers:

- Who is the target character?
- What item should be bought?
- What are the price, quantity, HQ, region, and routing constraints?
- Has a plugin claimed this intent?
- What is the server-visible lifecycle state?

The request is durable server state. It may outlive a plugin reload or failed local attempt.

The request is the user-facing identity. Retrying the same request three times is still one acquisition request, not three separate acquisition intents.

### Acquisition Claim

A claim is the server permission token that allows one plugin instance to act on one request.

It answers:

- Which plugin instance claimed the request?
- Which claim token authorizes lifecycle updates?
- Has the claim expired, been rejected, been cancelled, or been superseded?

The claim is not the route run. A claimed request can have zero, one, or multiple historical execution attempts over its lifetime.

### Execution Attempt

An execution attempt is one plugin-side try to fulfill one claimed request.

It answers:

- Which local attempt is active?
- What route stops were generated?
- What stop is being executed now?
- What sequence of lifecycle events has been emitted?
- What purchases, skips, and failures occurred during this attempt?

An attempt is identified by a server-visible `attemptId` generated by the plugin when the user starts or restarts execution. Restarting a failed route creates a new attempt id under the same request id. Reusing the same request id is expected; reusing the same attempt id after restart is not.

`runId` may still appear in code as an implementation name, but the product model should treat it as an execution attempt id, not as a new top-level acquisition object.

### Route Stop

A route stop is one world batch inside an execution attempt.

It answers:

- Which world is being visited?
- Which data center is it on?
- What listings did the advisory plan expect there?
- What live listings were actually observed?
- What was bought, skipped, exhausted, or failed there?

Route stops are ordered from the player's current world and data center outward, then by economic value inside those locality groups.

### Purchase Attempt

A purchase attempt is one attempt to buy one confirmed live listing.

It answers:

- Which live listing was selected?
- Which facts were revalidated immediately before purchase?
- Was purchase input sent?
- Did the listing disappear or otherwise prove success?
- If it failed, which failure class explains it?

Purchase attempts are the audit boundary for economic behavior.

### Run Event

A run event is a plugin-to-server progress or audit fact for one execution attempt.

It answers:

- Which request and attempt does this fact belong to?
- What sequence number is it?
- What phase or action happened?
- What stable payload describes it?

Run events are append-oriented. The server can project a latest request/route status from them, but the event stream itself should remain useful for diagnostics.

## Authority Boundaries

### Server Owns

- Request creation.
- Request queue status.
- Claim tokens and claim authorization.
- Dashboard-visible lifecycle projection.
- Durable lifecycle, diagnostic, and purchase summary records.
- Request cancellation or resend before active purchasing begins.

### Plugin Owns

- Local consent.
- Current character and current world detection.
- Advisory route preparation from server/external market data.
- Lifestream command dispatch.
- Market-board opening, search, listing reads, and purchase input.
- Live candidate validation.
- Current run state.
- Route stop progression.

### Dashboard Owns

- Creating acquisition intent.
- Showing queue state, latest run progress, and diagnostics.
- Cancelling or resending requests when server state allows it.

The dashboard does not directly drive game actions.

## Request Lifecycle

The request lifecycle is server-visible and durable.

```text
PendingPickup
  -> Claimed
  -> AcceptedInPlugin
  -> Running
  -> Complete
   | UnderProcured
   | Cancelled
   | Stopped
   | Failed
```

Additional server-side recovery states may be useful:

- `ClaimExpired`: claim was not accepted or used in time.
- `Superseded`: a newer run became authoritative for the same request.
- `StaleRunning`: server believes a run is active but no plugin heartbeat/progress has arrived inside the configured window.

These should be represented explicitly or as status reason fields, not inferred from vague failure strings.

## Plugin Attempt Lifecycle

The plugin attempt lifecycle is local execution state for one `attemptId`.

```text
NotStarted
  -> Starting
  -> Planning
  -> Ready
  -> Running
  -> Paused
  -> Stopping
  -> Complete
   | UnderProcured
   | Stopped
   | Failed
```

`Failed` means the attempt hit an unrecoverable or unknown condition. A failed attempt can be restarted, but restart creates a new `attemptId` and a fresh sequence stream under the same request.

`Stopped` means the user stopped local execution. It is not the same as server cancellation.

## Route Stop Lifecycle

Each route stop should be modeled independently.

```text
Pending
  -> Preflight
  -> TravelCommandSent
  -> Traveling
  -> Arrived
  -> MarketBoardOpening
  -> SearchingItem
  -> ListingsReady
  -> Purchasing
  -> Complete
   | Exhausted
   | Skipped
   | Failed
```

Expected transition states should not surface as errors:

- Character scope unavailable during data-center or world travel.
- Current world unavailable while on the title/menu transition path.
- Market-board windows closing before the next Lifestream command.
- Listing window closing after the last listing on a world is bought.
- Planned listing missing when other safe live candidates still exist.

They should be recorded as diagnostic events and classified as normal route progress.

## Purchase Attempt Lifecycle

```text
CandidateSelected
  -> Revalidating
  -> Revalidated
  -> PurchaseInputSent
  -> ConfirmationObserved
  -> PurchaseConfirmed
  -> Purchased
   | SkippedListingMissing
   | SkippedPriceChanged
   | SkippedBudgetExceeded
   | SkippedHqMismatch
   | InventoryFull
   | InsufficientGil
   | AmbiguousUi
   | Timeout
   | UnknownFailure
```

`UnknownFailure`, `AmbiguousUi`, inventory full, and insufficient gil stop the route. Listing disappearance, price drift above threshold, and HQ mismatch can skip the listing and rebuild the live candidate pool.

## Event Identity And Idempotency

Every lifecycle event sent from plugin to server should include:

- `requestId`
- `claimToken`
- `attemptId`
- `pluginInstanceId`
- `eventSequence`
- `eventType`
- `phase`
- `routeStopId`, when relevant
- `worldName`, when relevant
- stable payload JSON
- client timestamp UTC
- plugin visible version

The idempotency key should be deterministic per event:

```text
market-acquisition:{requestId}:{attemptId}:{eventSequence}:{eventType}
```

Rules:

- Same idempotency key and same request body returns the original response.
- Same idempotency key and different body is a client bug.
- A restart creates a new `attemptId`; it does not reuse an old attempt's event sequence.
- A retry of one event reuses the same key and same body.
- A later progress event uses a new sequence.
- The server should reject or ignore stale attempt events based on `attemptId`, not based only on request status.

The recurring 409 conflict shape means the current model is still allowing different facts to share the same idempotency identity. That should be treated as a model bug until proven otherwise.

## Conflict Policy

Not every conflict is equal.

### Real Conflicts

These should be logged loudly and fixed:

- Same idempotency key, different payload.
- Progress from an attempt id that the plugin believes is active but the server has never seen.
- Purchase audit for a request that was never accepted.
- Terminal status from a plugin without the matching claim token.

### Recoverable/Stale Conflicts

These should be represented as explicit stale or ignored outcomes:

- Old attempt reports after a newer attempt has started.
- Progress arrives after the request is already terminal with the same or newer attempt id.
- Route progress arrives after dashboard cancellation but before plugin sees the stop.

The server response should carry a structured reason such as `stale_attempt`, `request_terminal`, or `superseded_attempt`. The plugin should stop warning repeatedly once the stale condition is known.

## Reload And Restart Recovery

### Plugin Startup

On plugin startup or reload:

1. Load persisted accepted request and claim metadata.
2. Ask the server for the current request projection.
3. Reconcile local state:
   - `Complete`, `UnderProcured`, `Cancelled`, `Stopped`, or `Failed`: clear transient local route state, keep enough summary for display.
   - `PendingPickup` or `Claimed`: allow pickup/accept flow to continue if claim token is valid.
   - `AcceptedInPlugin`: allow prepare/start.
   - `Running`: if local run snapshot is complete enough, offer resume; otherwise require restart or stop.
4. Do not silently continue UI automation after reload.

### Route Restart

Restarting a failed or stopped route:

- keeps the same request id,
- keeps the same claim token if server allows it,
- creates a new attempt id,
- resets event sequence,
- rebuilds the advisory plan unless explicitly told to reuse the old one,
- clears stale stop/purchase local state.

### Dashboard Resend

Resend is for pickup recovery before active attempt execution.

Once a route has entered `Running` or `Purchasing`, dashboard controls should request local stop/cancel semantics instead of pretending the pending request can simply be resent.

## State Suppression And UI Visibility

The plugin UI should only show controls that are relevant to the current state.

- Hide request pickup while a route is actively running, pausing, stopping, or purchasing.
- Suppress "character scope unavailable" during expected travel/loading states.
- Suppress repeated server progress warning text after a known stale/superseded response.
- Show route controls only when a route exists.
- Show prepare controls only when an accepted request exists and no active route is running.
- Move raw input capture and live probe controls into diagnostics.
- Keep the main route surface compact: current phase, current world, next stop, latest purchase, and latest server sync.

## Diagnostics Model

Diagnostics should be persistent and correlated across client and server.

### Client Route Log

Each route log should record:

- request id,
- claim id/token hash or redacted token fingerprint,
- attempt id,
- plugin instance id,
- route nonce if retained,
- event sequence,
- idempotency key,
- route stop id,
- current world,
- expected world,
- visible tracked addons,
- automation precondition,
- automation postcondition,
- timeout/watchdog results,
- purchase attempt id,
- purchase result.

### Server Diagnostics

Server diagnostics should record:

- request id,
- attempt id,
- event sequence,
- idempotency key,
- event type,
- accepted/rejected/stale/superseded result,
- request status before and after,
- structured error reason,
- plugin version,
- server timestamp,
- sanitized payload summary.

The persistent dashboard diagnostics view should let us filter by request id, attempt id, event type, and severity. A 409 should be explainable from server diagnostics without needing to infer from the plugin log alone.

## Execution Invariants

- One plugin instance may have at most one active acquisition attempt.
- One request may have multiple historical attempts, but at most one active attempt.
- A route restart always means a new attempt id.
- A progress event's identity is request id plus attempt id plus sequence.
- A run event never mutates a different active attempt.
- Purchase execution never starts from remote data alone.
- Every purchase attempt is tied to a fresh live listing read.
- Every UI automation action has a precondition, postcondition, timeout, and diagnostic event.
- Terminal request states are durable and explicit.

## Immediate Implementation Direction

1. Introduce explicit attempt ids and event sequences in the plugin route runner/reporting path.
2. Add server-side run/event persistence or adapt `diagnostic_events` into a proper run event projection.
3. Change progress endpoints to reason about stale/superseded runs instead of returning generic transition conflicts.
4. Refactor route progress reporting out of `MainWindow` into a small run reporter service.
5. Add tests for:
   - restart creates a new attempt id,
   - duplicate same event is idempotent,
   - duplicate key with different body is rejected with structured detail,
   - stale old run event is ignored or classified without warning spam,
   - terminal request receives late progress without poisoning the active UI,
   - plugin suppresses scope warnings during world travel.
6. Update the plugin UI gating once the lifecycle states are explicit.
7. Add a dashboard/server diagnostics timeline for request/attempt/event inspection.

## Archive And Preset Boundary

Completed and failed requests should not be discarded from the dashboard just because they are terminal. They are useful as audit history and can later become reusable presets.

That is intentionally a feature layer, not a foundation requirement. The foundation only needs terminal requests and their attempt history to remain queryable. A later dashboard feature can expose:

- `Run again` from a completed or failed request.
- Saved request templates.
- Common item/price/region presets.
- Historical success/failure context for a preset.

When that feature lands, it should create a new request from the archived request's intent fields rather than mutating the old terminal request.

## Open Decisions

- Should a plugin reload during `Running` default to `Stopped`, `StaleRunning`, or `NeedsOperatorDecision` on the server?
- Should dashboard cancellation during `Purchasing` be a requested stop that the plugin must acknowledge, or a hard server terminal state?
- How long should run events and purchase attempts be retained compared to the existing 5,000 diagnostic event target?
- Should `UnderProcured` be a terminal request status, or a successful completion with a terminal reason?
- Should a failed run remain visible as retryable in the dashboard, or should retries always be initiated in the plugin after refetch/reconcile?

## Practical Next Step

The next coding pass should not add more ad hoc progress patches. It should make request identity, attempt identity, and event sequencing first-class, then change both plugin and server lifecycle reporting to use that model. That gives every later UI automation failure a place to land without creating a new state-sync problem.
